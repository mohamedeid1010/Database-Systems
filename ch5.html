01

عملية التصميم الفيزيائي

الـ Physical Database Design = نحوّل الوصف المنطقي (relations) لـ مواصفات تقنية لتخزين واسترجاع الداتا، بحيث الأداء يبقى كويس مع الحفاظ على الـ integrity والأمان والاسترجاع.

الغرض والهدف Purpose & Goal

Purpose: تترجم الوصف المنطقي للداتا لمواصفات تقنية للتخزين والاسترجاع.

Goal: تصميم لتخزين الداتا يدّي أداء كافٍ (performance) ويضمن الـ integrity والأمان (security) والاسترجاع (recoverability).

المدخلات الأساسية — سؤال 3 Inputs

أهم 3 مدخلات للتصميم الفيزيائي (احفظهم كده):

  • 1 الـ logical database structures اللي اتعملت في الـ logical design (الـ normalized relations).
  • 2 متطلبات معالجة المستخدم (user processing requirements) اللي اتحددت في الـ requirements definition.
  • 3 خصائص الـ DBMS ومكونات بيئة التكنولوجيا المستخدمة.
💡 تفصيل أكتر من السلايد: الـ inputs بتشمل كمان: تقديرات الحجم (volume estimates)، تعريفات الصفات، توقعات وقت الاستجابة، احتياجات الأمان والـ backup/recovery، وتوقعات الـ integrity.

القرارات الأساسية — سؤال 4 Key Decisions

القرارات اللي بتاخدها في التصميم الفيزيائي:

  • a اختيار صيغة التخزين (data type) لكل صفة — عشان تقلل المساحة وتعظّم الـ integrity.
  • b تجميع الصفات في physical records — مش دايماً نفس أعمدة الجدول المنطقي بتبقى أفضل تجميع.
  • c ترتيب الـ records في الـ secondary memory (file organizations) عشان التخزين/الاسترجاع/التحديث يبقى سريع + حماية واسترجاع الداتا.
  • d اختيار هياكل لربط الملفات (indexes & database architectures) عشان استرجاع الداتا المرتبطة يبقى أكفأ.
  • e تجهيز استراتيجيات للـ query optimization تستغل الـ file organizations والـ indexes.

02

تصميم الحقول — Designing Fields

الـ Field هو أصغر وحدة من داتا التطبيق بيتعرّف عليها الـ system software. تصميم الحقل = اختيار data type + الترميز/الضغط/التشفير + التحكم في الـ integrity.

اختيار الـ Data Type — سؤال 5 Objectives

الـ data type هو صيغة التخزين المعرّفة بالـ DBMS لكل صفة. أهداف اختياره (أهميتها بتختلف حسب التطبيق):

  • 1 تحسين الـ data integrity (تكامل الداتا).
  • 2 تمثيل كل القيم الممكنة (represent all possible values).
  • 3 دعم كل عمليات معالجة الداتا (support all data manipulations).
  • 4 تقليل مساحة التخزين (minimize storage space).

ليه بنرمّز القيم؟ Coding — سؤال 6 Coding

لما الصفة يكون ليها مجموعة قيم متفرّقة (sparse) أو حجمها كبير جداً وهتاكل مساحة، بنفكّر في الترميز (coding) أو الضغط (compression).

📌 الحقول الكبيرة معناها إن الداتا متباعدة أكتر → معالجة أبطأ. لو مجموعة القيم الصحيحة صغيرة، نترجمها لكود ياخد مساحة أقل. وتقنيات الضغط كمان بتستخدم الترميز لتقليل المساحة للأنماط المتكررة.
💡 من السلايد (Pine Valley): الكود بيوفّر مساحة، بس بيكلّف lookup إضافي في جدول الأكواد عشان تجيب القيمة الفعلية.

التحكم في تكامل الحقل Field Data Integrity

  • Default value القيمة المفترضة لو مفيش قيمة صريحة اتدخلت.
  • Range control حدود القيم المسموحة (constraints / validation rules).
  • Null value control السماح أو منع الحقول الفاضية.
  • Referential integrity الـ range control (والـ null) عشان الـ FK يطابق PK موجود.

الطرق الثلاثة — سؤال 7 Handling Missing Values

  • 1 Substitute حُط تقدير للقيمة الناقصة (مثلاً بمعادلة على متوسط القيم الموجودة). لازم القيم دي تتعلّم عشان المستخدم يعرف إنها مش حقيقية.

✅ ربط مهم: الـ Triggers (روتين بيشتغل أوتوماتيك لما حدث يقع أو وقت يعدّي) بتُستخدم لتنفيذ طرق التعامل دي.

الـ Denormalization

عكس الـ Normalization: بناخد relations صغيرة ونـدمجها في relations أكبر عشان الـ queries والتقارير تتعالج بـ overhead أقل (أسرع).

التعريف والفايدة Definition

الـ Denormalization: تحويل الـ normalized relations لمواصفات physical records غير منظّمة عشان الأداء.

الفايدة (Benefit)

بتحسّن الأداء (السرعة) عن طريق تقليل عدد الـ table lookups — يعني تقليل عدد الـ join queries اللازمة.

VS

التكلفة (Cost)

بسبب تكرار الداتا: إهدار مساحة تخزين + تهديد للـ integrity/consistency.

فرص الـ Denormalization الشائعة Opportunities

  • 1:1 علاقة one-to-one بين كيانين → ندمجهم في جدول واحد (Fig 5-2).
  • M:N + attrs علاقة many-to-many فيها non-key attributes (associative entity) → ندمج صفات أحد الكيانات في جدول العلاقة عشان نتجنّب join متكرر (Fig 5-3).
  • Reference data علاقة 1:N والطرف الـ "one" عنده داتا مش مستخدمة في أي علاقة تانية (Fig 5-4).
⚠️ Denormalize with caution: الـ denormalization ممكن (1) يزوّد فرص الأخطاء والتضارب، (2) يرجّع الـ anomalies تاني، (3) يجبرك على إعادة برمجة لما الـ business rules تتغير. وأحياناً فيه طرق أحسن لتحسين أداء الـ joins زي: تنظيم الجداول في الداتابيز (file organization & clustering)، وتصميم وتحسين الـ query صح.

ملفات الداتابيز الفيزيائية

إزاي الداتا بتتخزّن فعلياً على الـ secondary memory: physical files و tablespaces.

Physical File و Tablespace Storage

Tablespace: وحدة تخزين منطقية مسمّاة تقدر تخزّن فيها داتا من أكتر من جدول/view/object.

تنظيم الملفات — File Organizations

الـ File Organization = تقنية الترتيب الفيزيائي لـ records الملف على الـ secondary storage. 3 أنواع: Sequential · Indexed · Hashed.

عوامل اختيار الـ File Organization Factors

  • 1 استرجاع داتا سريع (fast retrieval & throughput).
  • 2 استغلال كفؤ لمساحة التخزين.
  • 3 الحماية من الأعطال وفقدان الداتا.
  • 4 تقليل الحاجة لإعادة التنظيم (reorganization).
  • 5 استيعاب النمو (growth).
  • 6 الأمان من الاستخدام غير المصرّح.

الـ Indexes والمفاتيح

الـ Index هو جدول/هيكل بيانات بيحدد مكان الـ records اللي بتحقق شرط في الملف. ده اللي بيخلّي الاسترجاع سريع.

استخدام واختيار المفاتيح Using Keys

Unique key index

على مفتاح فريد — بيفرض الـ uniqueness. مثال: CustomerID (PK) لـ Customer، أو composite PK لـ OrderLine.

Secondary key index

على حقل ممكن يتكرّر (مش فريد). مثال: حقل Description لـ Product.

الـ Join Index Join Index

index على أعمدة من جدولين أو أكتر جايين من نفس domain القيم → بيسرّع عمليات الـ join.

📌 نوعين: (أ) join index لأعمدة non-key مشتركة، (ب) join index لمطابقة foreign key (FK) مع primary key (PK).

قواعد استخدام الـ Indexes — سؤال مهم جداً Rules

  • 1 استخدمها على الجداول الكبيرة.
  • 2 اعمل index للـ primary key بتاع كل جدول.
  • 3 اعمل index لحقول البحث (اللي بتيجي كتير في WHERE).
  • 4 الحقول في ORDER BY و GROUP BY.
  • 5 لما يكون فيه أكتر من 100 قيمة مختلفة، ومش لما يكون أقل من 30.
  • 6 تجنّب الـ index للحقول ذات القيم الطويلة (يفضّل تضغطها الأول).
  • 7 لو المفتاح بيحدد مكان الـ record، استخدم surrogate (زي رقم تسلسلي) عشان التوزيع يبقى متساوي في مساحة التخزين.
  • 8 الـ DBMS ممكن يكون عنده حد لعدد الـ indexes لكل جدول وعدد البايتات للحقل المفهرس.
  • 9 خلّي بالك من فهرسة الحقول ذات الـ null — كتير من الـ DBMSs مش بتعترف بالـ null في بحث الـ index.

3 Major inputs into physical database design ›
What are the major inputs into physical database design?
Three major inputs to physical design:
  • a Logical database structures developed during logical design.
  • b User processing requirements identified during requirements definition.
  • c Characteristics for the DBMS and other components of the computing technology environment.
4 Key decisions in physical database design ›
What are the key decisions in physical database design?
Key decisions in physical database design:
  • a Choosing the storage format (called data type) for each attribute from the logical data model: the format is chosen to minimize storage space and to maximize data integrity.
  • b Grouping attributes from the logical data model into physical records: you will discover that although the columns of a relational table are a natural definition for the contents of a physical record, this is not always the most desirable grouping of attributes.
  • c Arranging similarly structured records in secondary memory (primarily hard disks) so that individual and groups of records can be stored, retrieved, and updated rapidly (called file organizations): consideration must also be given to protecting data and recovering data after errors are found.
  • d Selecting structures for storing and connecting files to make retrieving related data more efficient (called indexes and database architectures).
  • e Preparing strategies for handling queries against the database that will optimize performance and take advantage of the file organizations and indexes that you have specified: efficient database structures will be of benefit only if queries and the database management systems that handle those queries are tuned to intelligently use those structures.
5 Objectives of selecting a data type ›
What are the objectives of selecting a data type for a field?
These four objectives will have varying relative importance for different applications:
  • a Improve data integrity
  • b Represent all possible values
  • c Support all data manipulations
  • d Minimize storage space.
6 Why are field values sometimes coded? ›
Why are field values sometimes coded?
Coding or compressing field values:

Where attributes have a sparse set of values or a volume so large that considerable storage space will be consumed, possibilities for coding or compressing field values should be considered. Large data fields mean that data are further apart, which yields slower data processing. Where the set of valid values is small, translation into a code that requires less space is a possibility. Data compression techniques also use coding to reduce the storage space required for commonly recurring patterns of data.

7 Three ways to handle missing field values ›
Describe three ways to handle missing field values.
Three ways to handle missing field values:
  • a Substitute an estimate of the missing value: for example, for a missing sales value when computing monthly product sales, use a formula involving the mean of the existing monthly sales values for that product indexed by total sales for that month across all products. Such estimates must be marked so users know that these are not actual values.
  • b Track missing data so that special reports and other system elements cause people to quickly resolve unknown values. Setting up a trigger in the database can accomplish this. A trigger is a routine that will automatically execute when some event occurs or time period passes. One trigger could log the missing entry to a file when a null or other missing value is stored, and another trigger could run periodically to create a report of the contents of this log file.
  • c Perform sensitivity testing so that missing data are ignored unless knowing a value might significantly change results. For example, if total monthly sales for a particular salesperson were almost over a threshold that would make a difference in that person's compensation, then attention would be drawn to the missing value. Otherwise, it would be ignored. This is the most complex of the methods mentioned; it requires the most sophisticated processing logic, which is often implemented in application programs.

Consider the following normalized relations for a sports league. What recommendations would you make regarding opportunities for denormalization? What additional information would you need to make fully informed denormalization decisions?
Recommendations for denormalization, sports league:

One possibility for denormalization would be the inclusion of SpecialtyDescription attribute in the PLAYER relation.   • b What are the meaning and the contents of the ManagerTeam attribute in the MANAGER relation? Does this attribute refer to the TeamID attribute in the TEAM relation? Is a Manager instance also a Player instance?

التعريفات والمقارنات

أسئلة 1 (التعريفات) و 2 (المقارنات) بنص الشيت بالحرف — دي اللي بتيجي مضمونة.

سؤال 1 — Define each of the following terms Definitions

  • a. File organization A technique for physically arranging the records of a file on secondary storage devices.
  • b. Sequential file org. Records in the file are stored in sequence according to a primary key value.
  • c. Indexed file org. Records are either stored sequentially or non-sequentially, and an index is created that allows software to locate individual records.
  • d. Hashed file org. The address for each record is determined using a hashing algorithm.
  • e. Composite key A key made up of more than one column.
  • f. Secondary key One or a combination of fields for which more than one record may have the same combination of values.
  • g. Data type Each unit of a detailed coding scheme recognized by system software, such as a DBMS, for representing organizational data.
  • h. Join index An index on columns from two or more tables that come from the same domain of values.

سؤال 2 — Contrast the following terms Contrast

Physical file

A physical file is a named portion of secondary memory (magnetic tape, hard disk) allocated for the purpose of storing records.

vs

Tablespace

A tablespace is a named set of disk storage elements in which physical files for database tables may be stored.

Normalization

The process of breaking up larger relations with structural problems (i.e., anomalies) into smaller, well-structured relations.

vs

Denormalization

The opposite process of normalization: takes smaller relations and combines them into bigger relations to produce relations processed with less management overhead for queries, reporting, and sometimes data maintenance.

Secondary key

One field or a combination of fields for which more than one record may have the same combination of values.

vs

Primary key

One or a combination of fields for which every record has a unique value. Hence, the primary key is a unique identifier for a row.

🔑 خلاصة تقفّل بيها: الـ Physical design = data types (تقلل مساحة + تزوّد integrity) → denormalization (تدمج عشان تقلل joins بس بحذر) → file organizations (sequential / indexed / hashed) → indexes (PK + WHERE + ORDER BY + >100 قيمة) → query optimization. ركّز على قواعد الـ index وأنواع الـ file organization — دول اللي بييجوا أكتر.

بالتوفيق يا أبو عيد 💪 — مفتاح الشابتر ده: الأداء (performance). الـ index بيسرّع القراءة، والـ denormalization بيقلّل الـ joins — بس الاتنين ليهم ثمن (مساحة + integrity).
